	•	Introducción
	•	Propósito
El presente informe tiene como objetivo documentar la ejecución de pruebas funcionales realizadas sobre el sistema TEAMMATES. A través de este informe se deja constancia de los resultados obtenidos, los defectos identificados y el grado de cumplimiento de los requisitos evaluados, con el fin de proporcionar información útil para la validación de la calidad del producto.

	•	Alcance
Este proceso abarca todas las áreas del sistema TEAMMATES, tanto en su interfaz de usuario (frontend en Angular) como en su lógica de negocio (backend en Java). Las pruebas incluyen desde acciones básicas como iniciar sesión o enviar retroalimentación, hasta flujos más complejos como administrar cursos, crear sesiones de feedback o analizar resultados. También se consideran pruebas automatizadas que se ejecutan automáticamente cada vez que alguien sube un cambio al código, y pruebas manuales que se realizan para validar casos específicos o cuando se sospecha de errores.

	•	Referencias
	•	ISO/IEC/IEEE 29119: Estándar internacional para pruebas de software.
	•	Documentación oficial del sistema TEAMMATES
	•	Repositorio oficial de TEAMMATES: https://github.com/TEAMMATES/teammates

	•	Entorno de pruebas
Para asegurar que la plataforma TEAMMATES funcione correctamente y sin errores antes de ser utilizada por los usuarios finales, se estableció un entorno de pruebas específico. Este entorno simula el uso real del sistema, pero de manera controlada y segura, ejecutándose localmente en una computadora personal. Esto permite realizar pruebas detalladas sin interferir con los datos reales o afectar la experiencia de los usuarios activos en la plataforma.

	•	Configuración
	•	Configuración del Entorno:
Las pruebas se llevaron a cabo en una instalación local del sistema TEAMMATES, es decir, el proyecto fue clonado y ejecutado directamente en el entorno localhost. Esta configuración permite al equipo de pruebas interactuar con la plataforma tal como lo haría un usuario final, pero con la ventaja de poder controlar completamente los datos, flujos y condiciones de prueba. Al trabajar de manera local, se pueden ejecutar pruebas de forma repetitiva, rápida y sin depender de la infraestructura en la nube del sistema.
	•	Navegadores Utilizados: 
	•	Google Chrome: fue el navegador principal por su alto rendimiento y uso generalizado.
	•	Brave: elegido por su naturaleza basada en Chromium y su enfoque en privacidad  y bloqueo de contenido, ideal  para probar compatibilidad	 con funciones que pueden verse afectadas por extensiones.
	•	Opera GX: utilizado para verificar cómo se comporta la plataforma en un navegador moderno dirigido a usuarios exigentes, especialmente con características visuales  personalizadas.
	•	Sistemas Operativos Utilizados:
Las pruebas se ejecutaron en dos versiones comunes de sistemas operativos Windows, para garantizar una cobertura adecuada:
	•	Windows 11 Home Single Language:  utilizado en computadoras modernas de uso doméstico.
	•	Windows 10 Pro: común en entornos corporativos y educativos, ideal para validar estabilidad en equipos de trabajo.
	•	Herramientas de registro de pruebas manuales:
	•	Sistema TEAMMATES
	•	Hojas de cálculo de Google
	•	Herramientas de registro de pruebas automatizadas:
Para asegurar una mayor eficiencia y precisión en el proceso, se utilizaron herramientas de automatización que permitieron verificar comportamientos repetitivos o complejos sin necesidad de intervención manual constante:
	•	TestNG: framework utilizado para estructurar y ejecutar casos de prueba automatizados escritos en Java, especialmente útil para organizar pruebas de lógica de negocio del backend.
	•	Selenium: herramienta que permite simular la interacción de un usuario real con el navegador, automatizando clics, navegación, ingreso de datos, y verificación de mensajes o comportamientos visuales.
	•	Limitaciones
	•	El acceso a servicios externos (correo electrónico, autenticación de Google) está limitado a lo observable en la interfaz.
	•	Las pruebas no incluyen validaciones sobre la infraestructura subyacente (Google App Engine, Google Cloud Datastore).
	•	Las correcciones de errores no forman parte del alcance del equipo de pruebas, ya que no se tiene intervención en el código fuente.

	•	Estrategia de Métodos de Prueba Aplicados
Durante la ejecución de pruebas se aplicaron distintos métodos de validación para asegurar que el sistema se comporte correctamente bajo diferentes condiciones. Estos enfoques permiten evaluar no solo que una funcionalidad funcione, sino que también responda bien ante datos inesperados o situaciones límite. Los principales métodos aplicados fueron:
	•	Partición de Equivalencia: se dividieron los posibles datos de entrada en grupos válidos e inválidos para reducir la cantidad de pruebas necesarias sin perder cobertura funcional. Por ejemplo, se probaron diferentes tipos de usuarios (estudiante, instructor, invitado) para ver cómo responde el sistema.
	•	Análisis de Valores Límite: se probaron valores cercanos a los extremos permitidos, como límites de caracteres en campos de texto, fechas cercanas a hoy o campos vacíos, para observar si el sistema maneja correctamente estas situaciones.

	•	Validación de Entradas: se ingresaron datos mal formateados, incompletos o erróneos para confirmar que el sistema rechaza correctamente estas entradas con mensajes claros y sin comportamientos inesperados.

	•	Pruebas de Seguridad y Transición de Estados: se simularon acciones como intentar acceder a secciones sin estar autenticado o cambiar de roles, para verificar que el sistema mantenga una navegación segura y no permita acciones no autorizadas.

	•	Pruebas de Casos de Uso: se siguieron paso a paso los flujos principales del sistema (como crear un curso, enviar retroalimentación o consultar resultados) para asegurar que cada funcionalidad trabaje como se espera desde el punto de vista del usuario final.

	•	Tablas de Decisión: se organizaron combinaciones de condiciones y respuestas esperadas en tablas para validar correctamente reglas complejas del sistema, como permisos por rol o visibilidad de mensajes según configuraciones.

	•	Cobertura
	•	Conclusiones
Durante la ejecución de las pruebas, basadas en el catálogo de casos previamente diseñado, se evaluaron las funcionalidades más importantes del sistema TEAMMATES, abarcando tanto el frontend como el backend, así como aspectos de integración entre componentes y la persistencia de datos en la base.
Se ejecutaron un total de 156 casos de prueba, los cuales fueron distribuidos entre distintos módulos funcionales del sistema. Los resultados muestran que aproximadamente el 90% de los casos fueron exitosos, lo que refleja una buena estabilidad general de la aplicación y un cumplimiento adecuado de los requisitos funcionales definidos.
Sin embargo, se identificaron algunos problemas técnicos relacionados con el entorno de despliegue, los cuales dificultaron parcialmente la ejecución fluida de ciertas pruebas, generando retrasos y la necesidad de realizar reintentos en algunos casos. Estos incidentes no afectaron la validez de los resultados, pero sí ponen en evidencia la importancia de contar con un entorno de pruebas estable y correctamente configurado desde el inicio.
En general, el proceso permitió detectar y documentar defectos relevantes, mejorar la cobertura funcional y reforzar la comunicación entre los roles de QA y desarrollo, contribuyendo a la mejora continua de la calidad del sistema.
