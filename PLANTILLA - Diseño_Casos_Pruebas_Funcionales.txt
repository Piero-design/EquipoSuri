	•	Introducción
	•	Propósito
El propósito de este documento es definir y estructurar el diseño de casos de prueba para la plataforma TEAMMATES, con el fin de asegurar la calidad funcional del sistema mediante la aplicación de técnicas de prueba de caja negra. Se busca validar que las funcionalidades críticas operen conforme a los requerimientos establecidos desde la perspectiva del usuario final.

Entre los objetivos específicos se incluyen: establecer casos de prueba basados en los requisitos funcionales, aplicar técnicas como partición de equivalencia, análisis de valores límite, tablas de decisión y transición de estados, desarrollar una matriz de trazabilidad entre requisitos y pruebas, planificar el esfuerzo de testing, y proporcionar una guía de referencia para la ejecución, seguimiento y mejora continua del proceso de pruebas.

	•	Alcance
El alcance de este documento se limita al diseño de casos de prueba utilizando exclusivamente técnicas de caja negra aplicadas a los requerimientos funcionales identificados , son mostrados mas adelante. Se consideran únicamente cuatro tipos de pruebas: Partición de Equivalencia, Análisis de Valores Límite, Tablas de Decisión y Transición de Estados. No se incluyen pruebas de caja blanca, pruebas de rendimiento ni validaciones internas del código.

	•	Metodología de diseño
	•	Técnicas de Caja Negra utilizadas 
Para garantizar una cobertura eficaz de las funcionalidades críticas de TEAMMATES, se emplean las siguientes técnicas:
3.1.1.Partición de equivalencia
Justificación: Se utiliza la técnica de Partición de Equivalencia para validar las funcionalidades que involucran entradas estructuradas, como el registro de cursos, la creación de sesiones o la búsqueda de estudiantes. Esta técnica permite dividir los datos de entrada en clases válidas e inválidas, reduciendo así la cantidad de casos necesarios para lograr una cobertura significativa. Por ejemplo, al probar la creación de un curso, se considera como clase válida el ingreso de todos los campos correctamente, y como clase inválida, un Course ID duplicado. 
3.1.2. Análisis de Valores Límite
Justificación: El sistema TEAMMATES contiene múltiples campos con restricciones específicas en longitud, formato y lógica numérica (como nombre de usuario, correos electrónicos, longitud de sesiones, distribución de puntos, etc.). Por ello, se aplica la técnica de Análisis de Valores Límite para validar que dichos campos cumplan con sus restricciones justo en los extremos permitidos. Por ejemplo, se evalúa que un nombre de sesión con 65 caracteres sea rechazado si el límite es de 64. Esta técnica garantiza que el sistema reaccione correctamente frente a entradas críticas y evita vulnerabilidades comunes relacionadas con límites de validación.
3.1.3.Tabla de Decisión
Justificación: TEAMMATES permite configuraciones avanzadas como visibilidad de preguntas y asignación de puntos, lo cual depende de múltiples condiciones que interactúan. La técnica de Tabla de Decisión se aplica para modelar y verificar esas combinaciones de reglas. Por ejemplo, se valida que al seleccionar que tanto instructores como compañeros puedan ver una respuesta, se activen correctamente las opciones de visibilidad y se reflejen en la sesión. Esta técnica ayuda a representar y evaluar de manera sistemática decisiones complejas dentro del sistema, asegurando que cada combinación de opciones produzca el resultado esperado.

3.1.4.Transición de Estados
Justificación: En TEAMMATES, varias entidades (como sesiones de feedback o cursos) tienen estados definidos como “activa”, “archivada” o “eliminada”. La técnica de Transición de Estados se implementa para garantizar que los cambios de estado ocurran correctamente ante acciones del usuario. Por ejemplo, al eliminar lógicamente una sesión activa, esta debe desaparecer de la tabla de sesiones activas y aparecer en la de eliminadas. También se valida que, si se cancela la eliminación, la sesión conserve su estado original. Esta técnica es clave para verificar la coherencia en el flujo de trabajo y la integridad del ciclo de vida de los elementos.

	•	Casos de Prueba
4.1. Estructura de Casos de Prueba
Cada caso de prueba será descrito siguiendo una estructura estandarizada que facilita su comprensión, ejecución y trazabilidad. La estructura adoptada es la siguiente:
	•	ID: Código identificador único del caso de prueba (e.g., CP-001).
	•	Funcionalidad: Breve título que indique el propósito de la prueba.
	•	Descripción: Explicación clara y concisa del objetivo del caso de prueba.
	•	Requisito Asociado: Código o referencia del requerimiento funcional o no funcional asociado.
	•	Precondiciones: Condiciones que deben cumplirse antes de ejecutar la prueba (estado del sistema, usuario autenticado, datos necesarios, etc.).
	•	Datos de Entrada: Información o parámetros requeridos para ejecutar la prueba.
	•	Pasos de Ejecución: Secuencia detallada de acciones que debe seguir el evaluador para llevar a cabo la prueba.
	•	Técnicas de Pruebas: Técnica de caja negra utilizada para diseñar el caso.
	•	Prioridad: Nivel de importancia del caso (Alta, Media, Baja) según impacto y probabilidad.

4.2.21 Editar Respuesta

ID
CPF-0021
Funcionalidad
Editar respuesta (modo estudiante)
Descripción
Permite al estudiante modificar una respuesta previamente enviada dentro de una sesión activa. Se valida que el campo de texto no esté vacío, que no supere el límite de caracteres y que no contenga caracteres maliciosos como HTML o SQL.

Precondiciones
	•	Usuario autenticado como estudiante.

	•	Acceder a la sección “Sesiones de feedback”.
	•	Seleccionar la sesión con una respuesta editable.
	•	Hacer clic en “Editar respuesta”.
	•	Ingresar nuevo contenido en el campo de texto.
	•	Hacer clic en Guardar.
	•	Verificar que la respuesta haya sido actualizada y se muestre el mensaje de confirmación.

	•	Conclusiones
Las técnicas de caja negra y el enfoque por niveles de pruebas implementados en TEAMMATES permiten diseñar un plan de validación eficiente y exhaustivo, orientado a descubrir tanto errores de lógica en la gestión de datos como defectos en la interacción con el usuario. Al combinar partición de equivalencia, análisis de valores límite y tablas de decisión, se optimiza la selección de casos de prueba para cubrir amplios escenarios de entrada sin redundancias innecesarias. Las pruebas de transición de estados y los casos de uso proporcionan además una visión del comportamiento dinámico del sistema en situaciones reales, mientras que las validaciones de entrada, seguridad y usabilidad garantizan la robustez, la integridad de los datos y una experiencia de usuario coherente.
La estructuración en pruebas unitarias, de integración y de sistema dota al proyecto de un marco de calidad sólido: cada componente se verifica de forma aislada, sus interacciones se validan con precisión y el producto final se evalúa en condiciones simuladas de producción, incluida la respuesta ante cargas elevadas. Este enfoque jerárquico asegura detección temprana de regresiones, facilita el mantenimiento continuo del código y refuerza la confianza de los usuarios en la plataforma. En conjunto, la metodología propuesta ofrece un equilibrio entre cobertura, eficiencia y coste, alineándose con los objetivos de fiabilidad y escalabilidad de TEAMMATES.
